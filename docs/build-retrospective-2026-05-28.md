# QuizTube Build 复盘文档

## 1. 项目背景与目标

QuizTube 的核心目标是把 B 站视频转成可编辑的 Cornell 笔记，并进一步进入 Quiz 复习闭环。  
本轮建设重点是从“可演示原型”走向“可持续使用的 MVP”：

- 可靠转写（字幕优先 + ASR 兜底）
- 可感知的生成状态（前端任务中心）
- 可落库与可回放（笔记/Quiz 状态持久化）
- 可调度复习（会话答题 + 间隔复习）
- 可扩展到合集（多分P逻辑合并）

---

## 2. Build 过程时间线（按阶段）

### 阶段 A：ASR 与模型链路打通

- 初始目标：全流程跑通“视频 -> 转写 -> 笔记”。
- 关键动作：
  - 尝试 NVIDIA 方案（本地/托管）后转向云端替代
  - 统一为 OpenAI-compatible 调用模式，降低 provider 切换成本
  - 接入 Xiaomi MiMo（chat + transcribe）
- 结果：
  - 完成可切换的模型配置链路
  - 形成较清晰的错误提示（连接失败、404、不支持 endpoint）

### 阶段 B：性能与稳定性优化

- 关键动作：
  - 增加 `transcript_cache.json`（video_id + asr_model + language）
  - 增加音频分片转写（chunking）
  - 增加阶段耗时日志（下载/转写/总结/总耗时）
- 结果：
  - 重复请求显著加速
  - 大视频更稳，不易因单次大请求失败
  - 后续优化有了量化依据

### 阶段 C：前端体验从“等待”转向“可感知”

- 关键动作：
  - 全屏 loading 改为右上角任务中心
  - 增加进度、最小化、最近任务历史
  - 接入真实 B 站嵌入播放
  - Cornell 编辑区动态化（不再固定 2 条）
- 结果：
  - 生成过程更透明
  - 用户对“系统是否卡住”的焦虑下降

### 阶段 D：Quiz MVP 主链路实现

- 关键动作：
  - 新增 `quiz_store.json`，解耦 `notes.json`
  - 新增 `/api/quiz/*`（大厅、会话、答题、完成）
  - 实现简化间隔复习调度（1/2/4/8/16）
  - 题目生成接入、落库、状态回写
- 结果：
  - 从“静态 UI 原型”升级为“真实可作答闭环”
  - 核心指标（pending_quiz_count/status/next_review_at）可持续更新

### 阶段 E：速度与状态一致性修复

- 关键动作：
  - 把 Quiz 生成从主链路拆到后台（避免阻塞笔记返回）
  - 增加 `quiz_generation_status` + `quiz_needs_regen`
  - 补充“状态自愈”逻辑（避免卡在 generating）
- 结果：
  - 首屏生成速度提升
  - 状态展示与真实数据更一致

### 阶段 F：合集逻辑合并转写

- 关键动作：
  - 新增合集探测 API 与分P选择
  - 按分P顺序转写并做 timestamp 偏移
  - 生成单条聚合 note + 后台 Quiz
- 结果：
  - 支持“单链接覆盖多分P学习内容”
  - 成本可控（逻辑合并，不做物理拼接）

## 3. 关键问题与根因

### 问题 1：Provider 切换成本高

- 根因：各家“看似兼容”但细节不同（ASR endpoint、参数结构差异大）。
- 处理：统一抽象配置层 + provider 特例处理。

### 问题 2：慢体验主要来自同步串行链路

- 根因：笔记生成、Quiz 出题、转写等在单请求串行。
- 处理：把 Quiz 生成异步化，主链路先返回核心结果。

### 问题 3：状态容易“脏”

- 根因：异步任务完成后存在极少数回写遗漏，或历史数据不一致。
- 处理：读路径加 normalize/self-heal，保证 UI 最终一致。

### 问题 4：B 站风控和本地依赖导致不可用

- 根因：`yt-dlp` 缺失、cookies 未配置、验证码/412 风控。
- 处理：环境前置检查 + 错误文案改进 + 配置引导。

---

## 4. 本轮有效经验

- 先打通“可用闭环”，再做“体验优化”，收益更稳。
- 缓存 + 分片 + 日志是三件必须同时做的性能基础设施。
- 前端状态中心比“单纯 loading 动画”更能降低用户不确定感。
- 任何异步设计都要同时配“状态机 + 自愈逻辑”。
- 外部平台依赖（B站）必须设计显式降级和引导，不应默默失败。

---

## 5. 当前遗留与改进优先级

## P0（立即）

- 启动前依赖检查（`yt-dlp`、`ffmpeg`、cookies）与自检接口
- 统一错误码与前端提示（当前仍有细节不一致）
- 合集长任务进度从“模拟”升级为“真实阶段进度”

## P1（短期）

- 合集转写中间结果缓存（按 part 组合键）
- 合集模式下失败分P可视化重试（而非全量重跑）
- Quiz 题目质量守卫增强（schema 更严格 + 自动修复策略）

## P2（中期）

- 队列化后台任务（可恢复、可取消、可并发控制）
- 观测体系（成功率/平均耗时/失败原因分布）
- 多平台视频扩展策略（在 B 站稳定后再扩）

---

## 6. 对团队协作的建议

- 需求变更频繁时，先维护“稳定的状态模型”，再扩功能。
- 所有影响体验的改动都应附带“前后耗时对比”。
- 保持“后端数据真相单一来源”，前端只做最小推导。

---

## 7. 结论

本轮构建已完成从原型到 MVP 的关键跃迁：  
系统具备了真实生产向能力（转写、笔记、复习、会话、调度、合集支持）。  
下一阶段应聚焦“稳定性与可运营性”（依赖自检、真实进度、任务可靠性）而非继续堆叠新功能。
